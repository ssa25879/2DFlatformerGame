# Uni-Run Claude Code Collaboration Guide

이 문서는 `D:\work\Uni-Run` 프로젝트에서 Claude Code, Codex, 사용자 간 협업 시 따라야 할 통합 지침이다. 사용자에게 보이는 보고는 한국어로 작성하고, 내부 에이전트 간 작업 지시는 필요할 때 영어를 사용할 수 있다.

## 1. 프로젝트 기준

- 프로젝트 경로: `D:\work\Uni-Run`
- GitHub 원격 저장소: `https://github.com/ssa25879/2DFlatformerGame.git`
- 기본 브랜치: `main`
- 현재 프로젝트 목표: 기존 Unity 2D 러너 예제를 점프 없는 마우스 대시 기반 2D 플랫포머 프로토타입으로 전환하고 확장한다.
- 주요 씬: `Assets/Scene/RelicDash_ForestRuins_Stage01.unity`
- 보조/이전 씬: `Assets/Scene/Main.unity`

## 2. 사용자 응대 규칙

- 사용자에게 보고하거나 답변할 때는 한국어로 작성한다.
- 정보 제공은 필요한 내용만 간결하고 정확하게 한다.
- 변경 가능성이 있는 정보, 외부 서비스 상태, 최신 버전, 라이선스, GitHub 상태 등은 확인 후 답변한다.
- 사용자가 날짜나 상대 시점을 언급하면 필요 시 절대 날짜를 함께 적어 혼동을 줄인다.
- 사용자가 직접 수정한 내용은 되돌리지 않는다.
- 사용자에게 파일을 복사하거나 저장하라고 말하지 않는다. 같은 작업공간을 공유한다고 가정한다.

## 3. 파일 작업 및 백업

- 파일을 생성하거나 수정하기 전 기존 파일이 있으면 백업을 만든다.
- 새 파일은 이전 버전이 없으므로 백업 대상이 없음을 작업 기록에 남긴다.
- 백업은 프로젝트 안의 `Backups/` 또는 해당 영역의 `Backup/` 폴더를 사용한다.
- 백업 파일과 백업 폴더는 Git에 올리지 않는다.
- 기존 사용자 변경을 삭제, 되돌림, 재정렬하지 않는다.
- 불필요한 대규모 리팩터링이나 포맷 변경을 피한다.

## 4. 작업 로그

- 작업 로그는 `D:\Codex\Log\YYYYMMDD_Uni-Run_Log.md` 형식으로 작성한다.
- 작업일이 바뀌면 새 로그 파일을 만든다.
- 로그 파일은 최초 생성 이후 수정/추가만 한다.
- 로그 파일은 삭제하지 않는다. 삭제가 필요하면 사용자 확인을 먼저 받는다.
- 로그 파일 자체는 백업하지 않는다.
- 로그에는 수행한 작업, 생성/수정 파일, 백업 위치, 검증 결과, 실패/제약 사항을 간단히 남긴다.

## 5. 코딩 작업 원칙

- 작업 전 요구사항과 성공 기준을 짧게 정리한다.
- 불명확한 요구사항은 추측하지 말고 질문한다.
- 여러 해석이 가능하면 선택지를 제시한다.
- 진행 중 세운 가정은 숨기지 않고 명시적으로 밝힌다.
- 더 단순한 대안이 있으면 알리고, 근거가 있으면 요청에 반대 의견도 제시한다.
- 가장 단순한 구현으로 요청을 해결한다.
- 요청받지 않은 기능, 추상화, 설정 옵션을 추가하지 않는다.
- 한 번만 쓰는 코드를 위해 과한 구조를 만들지 않는다.
- 발생할 수 없는 상황에 대한 에러 처리나 방어 코드는 추가하지 않는다.
- 코드가 과하다고 느껴지면 "시니어 엔지니어가 봐도 과할까?"를 자문하고 단순화한다.
- 변경은 요청 범위에 직접 연결되는 줄로 제한한다.
- 주변 코드 스타일과 기존 패턴을 따른다.
- 내 변경으로 생긴 미사용 코드나 임포트는 정리한다.
- 기존의 무관한 죽은 코드나 오래된 구조는 임의로 삭제하지 않고 필요 시 보고만 한다.

## 6. 디버깅 원칙

- 증상을 바로 고치기 전에 원인을 확인한다.
- 오류 메시지, 콘솔 로그, Unity Editor 로그, 테스트 결과를 먼저 읽는다.
- 재현 조건과 최근 변경 사항을 확인한다.
- 비슷한 정상 동작 코드를 찾아 비교한다.
- 가설은 하나씩 세우고 최소 변경으로 검증한다.
- 세 번 이상 수정 시도가 실패하면 구조나 접근 자체를 재검토하고 사용자에게 상황을 보고한다.

## 7. 테스트 및 검증

- 기능 변경이나 버그 수정은 가능하면 먼저 실패하는 테스트 또는 재현 절차를 만든다.
- 다단계 작업은 시작 전 "단계 → 검증 방법" 형식의 짧은 계획을 제시한다. 예: `1. [단계] → 검증: [확인 방법]`
- 성공 기준은 검증 가능한 형태로 정의한다(예: "동작하게 하기"가 아니라 "재현 테스트 작성 후 통과").
- Unity C# 파일 수정 후에는 컴파일 오류를 확인한다.
- 관련 EditMode/PlayMode 테스트가 있으면 실행한다.
- 자동 검증이 타임아웃되거나 도구 문제로 실패하면, 성공으로 말하지 말고 실제 상태를 그대로 보고한다.
- 완료 보고 전에는 방금 실행한 검증 결과를 확인한다.
- 검증 명령의 핵심 결과, 실패 수, 타임아웃 여부를 사용자에게 요약한다.

## 8. Unity 작업 규칙

- `Assets/`, `Packages/`, `ProjectSettings/`를 Unity 프로젝트의 핵심 추적 대상으로 본다.
- `Library/`, `Temp/`, `Logs/`, `UserSettings/`, 빌드 산출물, 백업은 Git에 포함하지 않는다.
- 플랫폼 렌더링, Sorting Layer, 플레이어 크기, 콜라이더 등 사용자가 직접 조정한 영역은 함부로 재구성하지 않는다.
- 씬 자동화 스크립트가 활성 씬을 바꿀 수 있으므로 작업 전후 현재 씬을 확인한다.
- `RelicDash_ForestRuins_Stage01.unity`와 `Main.unity`를 혼동하지 않는다.
- 같은 프로젝트의 Unity `AssetImportWorker` 프로세스는 별도 프로젝트 창으로 오해하지 않는다.
- Unity MCP나 batchmode가 컴파일/리로드 중 타임아웃될 수 있으므로 결과 파일과 Editor 로그를 함께 확인한다.
- 일반 Unity CLI `-runTests`가 결과 XML을 만들지 않는 경우 `CodexEditModeTestRunner.RunEditModeTests`를 사용한다.
- 스크립트 변경 직후 batchmode는 컴파일만 하고 끝날 수 있으므로 테스트 실행 메서드는 필요 시 다시 실행한다.

## 9. 현재 게임 구현 요약

- 점프는 제거되어 있다.
- 좌우 이동이 구현되어 있다.
- 좌클릭/우클릭으로 마우스 방향 대시가 동작한다.
- Space 대시는 동작하지 않는다.
- 공중 2연속 대시는 방지되어 있다.
- 착지, 플랫폼 위 아래 방향 대시 후 착지, DashRecharge Pickup 획득으로 대시가 충전된다.
- DashRecharge Pickup은 획득 후 사라지고 5초 뒤 같은 위치에 재생성된다.
- Goal 도달 시 클리어, DeadZone/Trap 접촉 시 게임오버가 동작한다.
- 게임오버/클리어 후 좌클릭, 우클릭, Space로 재시작할 수 있다.
- Dash Ready UI, 대시 피드백, 대시 예고선, UI 이미지 비율 보정 작업이 진행되어 있다.

## 10. UI 및 디자인 규칙

- UI를 만들거나 수정할 때는 사람이 보기에 괜찮은지, 배경/게임 톤과 어울리는지 확인한다.
- 접근성을 우선한다: 명확한 포커스 상태, 충분한 대비, 44x44px 이상의 인터랙션 타깃을 고려한다.
- 중요한 UI는 명확한 패널/카드 안에 배치한다.
- 하나의 주요 강조색과 그레이스케일 중심으로 구성한다.
- UI 텍스트/표면에 순수 검정 `#000000`을 쓰지 않는다.
- Unity UI에서는 공통 팔레트, prefab, ScriptableObject, 문서화된 테마 값 등 재사용 가능한 토큰을 우선한다.
- 그림자와 강조 효과는 절제한다.
- 24px 계열의 일정한 간격 리듬을 유지한다.
- 정보 표면에는 상태를 고려한다: 비어 있음, 로딩, 오류, 성공.
- 진행률/게이지에는 맥락이 되는 라벨이나 숫자를 제공한다.
- 폼은 가능하면 단일 컬럼, 상단 라벨, 복구 가능한 오류 메시지로 구성한다.
- 모션은 짧고 일관되며 상호작용 이해에 도움이 될 때만 사용한다.
- Unity에서는 Canvas, Panel, Prefab, TextMeshPro 스타일, Sprite, Material, Serialized Color를 UI 토큰/컴포넌트로 취급한다.
- UI 이미지가 찌그러지지 않도록 아이콘류는 `preserveAspect`를 확인한다.
- 텍스트가 버튼, 패널, 카드 밖으로 넘치거나 다른 요소를 가리지 않게 한다.

## 11. Git 및 GitHub 관리

- 원격 저장소는 `origin`으로 `https://github.com/ssa25879/2DFlatformerGame.git`를 사용한다.
- 커밋 전 `git status --short --branch`로 변경 범위를 확인한다.
- 무관한 사용자 변경은 스테이징하지 않는다.
- 여러 작업이 섞여 있으면 사용자에게 커밋 범위를 확인한다.
- `.gitignore`에 의해 다음은 업로드하지 않는다:
  - `Library/`
  - `Temp/`
  - `Logs/`
  - `UserSettings/`
  - `Backups/`, `Backup/`
  - `.codex/`, `.agents/`, `.vscode/`
  - `*.csproj`, `*.sln`, `*.slnx`
  - 키, 토큰, 인증서, `.env` 계열 파일
- GitHub 업로드 전 100MB 초과 파일과 민감 파일 후보를 확인한다.
- `gh` CLI가 없는 환경에서는 PR 생성 워크플로를 사용하지 않고 일반 `git push`로 처리한다.

## 12. 권한과 안전

- 파괴적 명령은 사용자 승인 없이 실행하지 않는다.
- `git reset --hard`, `git checkout --`, 대량 삭제, 재귀 삭제는 명시 요청이 있을 때만 수행한다.
- Windows에서 삭제/이동 작업을 할 때는 대상 경로가 프로젝트 내부인지 확인한다.
- 파일 쓰기 권한이 필요한 경우 승인 요청을 사용한다.
- 외부 네트워크, GitHub 푸시, GUI 실행, 샌드박스 밖 쓰기는 필요한 경우 사용자 승인을 받는다.

## 13. 협업 방식

- 진행 중에는 짧게 현재 하는 일과 이유를 공유한다.
- 긴 작업은 단계별 체크리스트로 상태를 갱신한다.
- 사용자가 새 메시지로 방향을 바꾸면 최신 요청을 우선한다.
- 작업 완료 후에는 변경 사항, 검증 결과, 미해결 사항을 짧게 보고한다.
- 성공했다고 말할 때는 방금 확인한 근거를 함께 제시한다.
